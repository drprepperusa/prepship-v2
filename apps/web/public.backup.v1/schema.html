Domain → DTO Transformation Logic

Implemented in /apps/api/src/modules/orders/application/list-orders.ts via toOrderDto() function

shipTo Object (Nested)
Combines 4 flat address fields: OrderRecord.shipToName OrderRecord.shipToCity OrderRecord.shipToState OrderRecord.shipToPostalCode

Into structured object: { name, city, state, postalCode }

Why: Frontend expects nested address structure; matches V1 API contract
weight Object (Nested)
Transforms single value: OrderRecord.weightValue: number

Into structured object: { value: number, units: "lbs" }

Why: Frontend expects units field; allows future metric conversion without schema change
label Object (Nested)
Combines 8 label-related fields: labelShipmentId, labelTracking, labelCarrier, labelService, labelProvider, labelCost, labelRawCost, labelShipDate

Into structured object: { shipmentId, trackingNumber, carrierCode, serviceCode, shippingProviderId, cost, rawCost, shipDate }

Why: Groups related label data for cleaner API contract
JSON Parsing
3 fields require JSON.parse():
OrderRecord.bestRateJson → OrderSummaryDto.bestRate
OrderRecord.selectedRateJson → OrderSummaryDto.selectedRate
OrderRecord.raw → OrderSummaryDto.raw
OrderRecord.items → OrderSummaryDto.items

Why: Database stores JSON as strings; API returns parsed objects

Key Relationships & Rules

Client → Orders
1:N relationship. Each client has many orders. Enforced via orders.clientId foreign key.
Orders → Shipments
1:N relationship. Each order may have multiple shipments. Tracked via shipments.orderId.
Order → OrderLocal
1:1 relationship. Extended metadata stored separately. order_local.orderId is FK+PK.
Inventory → Ledger
1:N relationship. Ledger tracks all quantity adjustments per SKU. inventory_ledger.invSkuId FK.
Ledger ↔ Orders
Optional relationship. Ledger entries may reference an order (nullable inventory_ledger.orderId).
Data Consistency
All clientId refs must exist in clients table. inventory_skus always scoped to client (multi-tenant).
📌 Data Consistency Rules:
• All clientId references must exist in clients table
• storeId in orders must match one of the client's storeIds array
• inventory_skus are always scoped to a client (multi-tenant isolation)
• inventory_ledger.orderId is nullable (stock adjustments may not be tied to orders)
• order_local must have matching orderId in orders (enforced via constraints)
• JSON fields (raw, items, bestRateJson, selectedRateJson) are stored as strings and parsed on API response
