Key Relationships

Client → Orders
1:N - Each client has many orders. Enforced via orders.clientId foreign key.
Orders → Shipments
1:N - Each order may have multiple shipments. Tracked via shipments.orderId.
Order → OrderLocal
1:1 - Extended metadata stored separately. order_local.orderId is FK+PK.
Inventory → Ledger
1:N - Ledger tracks all quantity adjustments per SKU. inventory_ledger.invSkuId FK.
Ledger ↔ Orders
Optional - Ledger entries may reference an order (nullable inventory_ledger.orderId).
SKU Bundles
inventory_skus.parentSkuId creates bundle hierarchies. Self-referential or cross-reference model.
📌 Data Consistency Rules:
• All clientId references must exist in clients table
• storeId in orders must match one of the client's storeIds array
• inventory_skus are always scoped to a client (multi-tenant isolation)
• inventory_ledger.orderId is nullable (stock adjustments may not be tied to orders)
• order_local must have matching orderId in orders (enforced via constraints)
